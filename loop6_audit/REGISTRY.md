### B-02 — SECURITY DEFINER functions missing SET search_path

- **Severity:** P1
- **Surface:** db
- **Lens findings merged:** A1-02, A1-17, A5-04
- **Root cause:** 10 SECURITY DEFINER functions have `proconfig IS NULL` (no `SET search_path`). The full list per A1-02: `add_credits`, `admin_count_profiles`, `admin_list_profiles`, `check_balance`, `deduct_credits`, `get_stripe_customer_id`, `get_user_tokens`, `handle_new_user`, `update_profile_by_id`, `update_profile_by_stripe_customer`. Because `pg_temp` is prepended to the search path by default, a caller can `CREATE TEMP TABLE profiles (...)` with a poisoned schema and shadow the real table during function execution. `handle_new_user` (A1-17) fires on signup and is particularly sensitive.
- **Fix sketch:**
  - For every SECURITY DEFINER function: add `SET search_path = ''` to the `CREATE OR REPLACE FUNCTION` declaration.
  - Fully qualify every table and function reference inside the body: `public.profiles`, `public.credit_buckets`, `auth.uid()`, etc.
  - File: new migration in `frontend/supabase/migrations/005_search_path_hardening.sql`.
  - Evidence: `pg_catalog` query `SELECT proname, proconfig FROM pg_proc ... WHERE prosecdef=true` returns `proconfig IS NULL` for all 10 listed functions (A1-02 lines: `proconfig IS NULL for: add_credits, admin_count_profiles, ...`).
- **Test to add:** `tests/contracts/C10_search_path_isolation.py` — for each definer function, create a TEMP TABLE shadow with the same name, invoke the function, assert the real `public` table is unchanged.
- **Blocking dependencies:** B-01 (REVOKE grants before hardening; the two migrations can be batched in the same file)
- **Confidence:** high
- **Notes:** A5-04 provides the adversarial reproduction confirming the attack is real, not hypothetical. A1-17 is specifically for `handle_new_user`; included here because it shares the same fix (add `SET search_path = ''`). B-14 (handle_new_user trigger idempotency) depends on this landing first. Can be batched in one migration with B-01's REVOKE block.

### B-03 — Stripe webhook marks event processed before business logic succeeds

- **Status:** FIXED 2026-04-27 — schema upgrade in `mariana/data/db.py`: `stripe_webhook_events` now carries `status` (pending|completed), `attempts`, `received_at`, `last_attempt_at`, `completed_at`, `last_error` columns (idempotent ALTER block for existing deployments). Webhook handler claims pending before business logic, finalizes only after handler returns; on raise, row stays pending and Stripe retries trigger RETRY claim. Per-grant idempotency via `uq_credit_tx_idem` (`ref_id=event_id`) prevents double-credit on retry. Backwards-compat shim retains old `_record_webhook_event_once`. New regression suite `tests/test_b03_webhook_two_phase.py` (8 tests covering NEW/RETRY/DUPLICATE/finalize-fail). Full suite: 7 SQL contracts + 66 pytest + 24 vitest GREEN.
- **Severity:** P1
- **Surface:** api
- **Lens findings merged:** A2-01
- **Root cause:** In `api.py` lines 5262–5273, `_record_webhook_event_once(evt_id, type)` is called and commits a `webhook_events` row before any credit-granting or subscription-update logic runs. - **Fix sketch:**
  - Move `_record_webhook_event_once` to **after** the business-logic block, within the same transaction, or use a two-phase approach: mark as `pending` before, `complete` after.
  - File: `/home/user/workspace/mariana/mariana/api.py` lines 5262–5273, 5301–5317, 6001–6011.
  - Alternatively, wrap the entire handler in a transaction: `async with db.transaction(): <business logic> <record event>`.

- **Blocking dependencies:** none
- **Confidence:** high

### B-06 — admin_set_credits absolute write races concurrent spend

- **Severity:** P1
- **Surface:** db
- **Lens findings merged:** A5-02
- **Root cause:** `admin_set_credits` reads `profiles.tokens` without `SELECT ... FOR UPDATE`, then writes the absolute new balance. A concurrent `spend_credits` that reads the old balance and commits after `admin_set_credits` silently overwrites the admin's intended value (or vice versa). The `previous_balance` captured in `admin_audit_log` is a dirty read. Evidence: pg_catalog prosrc for `admin_set_credits` + reproduction via two interleaved psql sessions (A5-02).
- **Fix sketch:**
  - Add `SELECT tokens INTO v_current FROM public.profiles WHERE id = p_user_id FOR UPDATE;` at the top of `admin_set_credits` body.
  - Insert a balancing `credit_transactions` row (`type='admin_adjust'`, `credits = new_balance - v_current`) so the ledger reflects the correction.
  - Log `previous_balance = v_current` (from the locked read, not the pre-lock snapshot) in `admin_audit_log`.
  - File: migration on `public.admin_set_credits`; batched with `005_ledger_repair.sql`.
- **Test to add:** `tests/contracts/C08_admin_spend_race.py` — interleave `spend_credits` and `admin_set_credits` via two asyncpg connections using savepoints; assert final `profiles.tokens` matches a deterministic ordering and ledger sums equal the balance.
- **Blocking dependencies:** B-01 (REVOKE grants first; admin_set_credits must be tightened before row-locking)
- **Confidence:** high
- **Notes:** Same-family as B-07 (spend race) — both require `SELECT ... FOR UPDATE`. Can be applied in the same migration. A5-02 explicitly lists `blocking: [A5-01]`.

### B-07 — spend_credits no SELECT FOR UPDATE — concurrent spend underflows balance

- **Severity:** P1
- **Surface:** db
- **Lens findings merged:** A1-09, A5-10
- **Root cause:** `spend_credits` body reads `profiles.tokens` into a local variable, then iterates `credit_buckets`, then writes back. There is no `SELECT ... FOR UPDATE` on the `profiles` row. Two concurrent calls both read the same `tokens=1000`, both write `tokens=400` — one spend is silently lost while the ledger records both debits (`sum(ledger.spend)=1200` vs `profiles.tokens=400`). Additionally, `balance_after` in the per-bucket loop reflects a partial mid-loop snapshot, not the final post-all-buckets balance (A1-09). Evidence: pg_catalog prosrc for `spend_credits`; local two-session reproduction in A5-10.
- **Fix sketch:**
  - Add `SELECT tokens INTO v_current FROM public.profiles WHERE id = p_user_id FOR UPDATE;` at the start of `spend_credits`.
  - `IF v_current < p_amount THEN RAISE EXCEPTION 'insufficient_credits';` before any bucket writes.
  - Compute `balance_after` once, after all bucket iterations complete, and store that single value in all `credit_transactions` rows for this call.
  - File: migration on `public.spend_credits`; batched in `005_ledger_repair.sql`.
- **Test to add:** `tests/contracts/C14_concurrent_spend.py` — fire two `spend_credits(user, 600)` in parallel via `asyncio.gather` starting from `tokens=1000`; assert exactly one succeeds, one raises `insufficient_credits`, and ledger sum equals `profiles.tokens` delta.
- **Blocking dependencies:** B-01 (anon execute revoked first)
- **Confidence:** high
- **Notes:** Same-family as B-06 (admin_set_credits race) and B-05 (ledger bypass). All three share the `005_ledger_repair.sql` migration. A5-10 lists `blocking: [A5-01]`. A1-09 `balance_after` corruption is a sub-issue; addressed in same fix.
