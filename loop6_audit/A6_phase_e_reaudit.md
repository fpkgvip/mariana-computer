### F-01 — Public preview route bypasses ownership checks
- **Severity:** P1
- **Category:** C
- **Surface:** api
- **Reproduction:**
  1. `GET /api/preview/{task_id}` is owner-gated: `mariana/api.py:1560-1579` loads `_deft_manifest.json`, reads `user_id`, and rejects non-owners with 403.
  2. `GET /preview/{task_id}/{file_path}` is not owner-gated: `mariana/api.py:1523-1558` validates only path shape and serves any file under `${DEFT_PREVIEW_ROOT}/{task_id}`.
  3. Preview ownership metadata definitely exists because the deploy path writes `{"task_id", "user_id", ...}` into `_deft_manifest.json` at `mariana/agent/dispatcher.py:583-593`.
  4. Result: anyone who learns another user's `task_id` can request `/preview/<task_id>/index.html` or nested assets directly and bypass the owner check that the manifest route enforces.
- **Root cause:** authorization is implemented only on the manifest-discovery endpoint, not on the static-file route that actually serves the preview.
- **Proposed fix:** require auth on `/preview/*` itself and enforce the same manifest-owner check before serving any asset, or replace raw task-id preview URLs with short-lived signed URLs/tokens. Test with two users: owner gets 200 for `/preview/{task_id}/index.html`; non-owner with a known task ID must get 403 for both the entry document and child assets.
- **Confidence:** high
- **Overlap with existing canonical (if any):** none

### F-02 — Start-investigation races pending-upload consumption and can double-charge
- **Severity:** P1
- **Category:** A
- **Surface:** cross
- **Reproduction:**
  1. Pending-upload ownership binding is serialized under `_get_upload_lock(f"pending-{normalized_session_uuid}")` in `mariana/api.py:4537-4545`.
  2. Investigation creation reserves credits before consuming uploads in `mariana/api.py:2671-2687`.
  3. The same investigation path then moves files out of `uploads/pending/{session_uuid}` without taking that pending-session lock in `mariana/api.py:2707-2740`, and only afterward persists the task in `mariana/api.py:2790-2807`.
  4. Two concurrent `POST /api/investigations` requests with the same `upload_session_uuid` can therefore both reserve credits, both pass the `.owner` check, and then race on `shutil.move(...)` / `rmdir()`: one task gets the files, the other gets partial or empty attachments, yet both tasks can be created.
- **Root cause:** the upload session is protected during upload-time writes, but not during submit-time consumption. The “claim session + move files + create task” sequence is not atomic.
- **Proposed fix:** reuse the same `pending-{session_uuid}` lock inside `start_investigation` and hold it across (a) session ownership check, (b) single-use claim/consume marker, (c) file move, and (d) task creation / credit rollback on failure. Better still, atomically rename the pending directory to a claimed path before moving files. Add a concurrency test that fires two investigation-start requests against one session UUID and asserts exactly one succeeds, exactly one reservation remains, and files are attached exactly once.
- **Confidence:** high
- **Overlap with existing canonical (if any):** none

### F-03 — Refund reversal forgives credits already spent before the clawback lands
- **Severity:** P1
- **Category:** F
- **Surface:** cross
- **Reproduction:**
  1. The reversal handler computes the intended debit from the original grant in `mariana/api.py:5905-5915` and sends that amount to the refund RPC in `mariana/api.py:5937-5945`.
  2. The SQL repair intentionally clamps the debit to current balance when the user has already spent the credits: `frontend/supabase/migrations/006_refund_credits_repair.sql:63-69`.
  3. The dispute path explicitly documents that a later definitive chargeback may return `no_credits` if earlier reversal logic already exhausted what was left, at `mariana/api.py:6013-6018`.
  4. Result: if a user receives credits, spends them, and only then gets refunded/charged back, the system drains only whatever balance is still present and silently forgives the rest.
- **Root cause:** the B-04 repair models reversal as “best-effort debit of currently available buckets” rather than “full economic clawback of the original grant.”
- **Proposed fix:** record the full clawback even when available buckets are insufficient. Concretely, introduce a debt/negative-balance construct or a locked deficit bucket that future grants/top-ups must satisfy before new spend is allowed. Regression test: grant 1000 credits, spend 900, then process a full refund; the account should retain a 900-credit deficit or equivalent blocked state, not emerge clean at zero.
- **Confidence:** high
- **Overlap with existing canonical (if any):** B-04 (related post-fix semantic gap, not the original unhandled-event bug)

### F-04 — Subscription downgrade/cancel never updates the plan field used for enforcement
- **Severity:** P1
- **Category:** F
- **Surface:** cross
- **Reproduction:**
  1. Investigation gating reads `profiles.plan` via `/profiles?select=plan` and enforces allowed tiers, duration caps, and continuous-mode access from that value in `mariana/api.py:2571-2629`.
  2. The profile schema still defines `plan` as a persistent entitlement column in `frontend/supabase/migrations/001_initial_schema.sql:1-9`.
  3. Stripe webhook handlers update only `subscription_plan`, `subscription_status`, and `subscription_current_period_end` in `mariana/api.py:5466-5478`, `5562-5569`, `5725-5734`, and `5761-5764`.
  4. The available profile-update RPCs can mutate `plan` only when payload includes that field (`frontend/supabase/migrations/007_loop6_b02_b05_b06_ledger_sync.sql:170-183`), but the webhook-by-customer helper used by those handlers does not set `plan` at all (`frontend/supabase/migrations/007_loop6_b02_b05_b06_ledger_sync.sql:194-203`).
  5. There is no migration in the repo that retires `plan` or rewires enforcement to `subscription_plan`.
- **Root cause:** subscription state and entitlement enforcement diverged into two separate columns. Webhooks maintain `subscription_*`, but the server still authorizes premium behavior from legacy `plan`.
- **Proposed fix:** choose one canonical entitlement field. Either (a) move all enforcement to `subscription_plan` + `subscription_status`, or (b) have every subscription webhook update `plan` transactionally alongside `subscription_*`. Add a downgrade regression test: start as `pro`, deliver `customer.subscription.deleted` or a downgrade event, then assert a subsequent deep-tier or continuous-mode investigation is rejected.
- **Confidence:** high
- **Overlap with existing canonical (if any):** none

### F-05 — Deleting auth.users leaves orphaned investigations and task artifacts behind
- **Severity:** P2
- **Category:** B
- **Surface:** cross
- **Reproduction:**
  1. `profiles.id` cascades from `auth.users(id)` in `frontend/supabase/migrations/001_initial_schema.sql:1-9`.
  2. Ledger tables also cascade from `auth.users`, while `stripe_events.user_id` is nulled on delete in `frontend/supabase/migrations/002_deft_credit_ledger.sql:84-91`.
  3. `research_tasks` has no foreign key to `auth.users` or `profiles`; ownership exists only inside `metadata JSONB` in `mariana/data/db.py:104-124`.
  4. API owner checks throughout the backend read `metadata.user_id`, not a relational FK, for example in `mariana/api.py:1229-1255` and `3017-3059`.
  5. Result: deleting a user from Supabase removes profile/ledger rows but leaves `research_tasks` and all task-linked children intact until manually deleted. Those rows become permanently orphaned because the owner record is gone.
- **Root cause:** the billing/profile domain is relationally attached to `auth.users`, but the investigation domain stores owner identity as unindexed JSON metadata instead of an FK.
- **Proposed fix:** add a first-class `user_id` FK on `research_tasks` (and backfill from metadata), then cascade deletes from there. Keep metadata as derived/cache data only. Regression test: create a user with a task and children, delete the auth user, and assert tasks, files, previews, and all task-linked intelligence rows disappear consistently.
- **Confidence:** high
- **Overlap with existing canonical (if any):** B-15 (adjacent FK/cascade family, but distinct orphan path)
